# Конституция VertexFlow Calculator

## Основные принципы

### I. Клиентская сторона

Вся функциональность калькулятора ДОЛЖНА выполняться на 100% на стороне клиента в браузере. Никакие серверные вычисления, хранение данных или вызовы API не допускаются. Этот принцип является НЕОСПОРИМЫМ.

**Правила**:
- Все вычисления, управление состоянием, валидация и рендеринг UI происходят исключительно в JavaScript
- Никаких fetch/axios HTTP-запросов к любому серверному эндпоинту
- Никаких cookies, сессий или серверного хранения состояния
- Никаких вызовов WordPress REST API для операций калькулятора
- Никакого хранения данных на бэкенде — всё находится в памяти браузера
- Ввод/вывод файлов ограничен FileReader API (импорт) и Blob/URL API (экспорт)

**Обоснование**: Устраняет серверные вычислительные затраты, обеспечивает мгновенный отклик без задержек сети, гарантирует, что данные пользователя никогда не покидают их браузер, и позволяет работать в офлайн-режиме с кэшированными ресурсами.

### II. Точность формата GLC

Калькулятор ДОЛЖЕН поддерживать двунаправленную обработку GLC-файлов без потерь, как определено в `.docs/GLC_FORMAL_SPECIFICATION/`. Операции импорта и экспорта ДОЛЖНЫ сохранять все геометрические и производственные данные.

**Правила**:
- Импорт: Парсинг GLC-файлов с кодировкой windows-1251 (без BOM), окончания строк CRLF
- Экспорт: Генерация GLC-файлов с идентичными требованиями к кодировке и структуре
- Сохранение всей геометрии: точки, дуги, вырезы, зоны с полной точностью координат
- Поддержка многкомнатных файлов-контейнеров с правильными блоками ROOMBEGIN/ROOMEND
- Сохранение всей числовой точности (значения FLOAT с десятичными точками в GLC, запятые в метрических экспортах)
- Валидация структурной целостности перед операциями экспорта

**Обоснование**: GLC является отраслевым стандартом для CAD/расчёта натяжных потолков. Конвертация без потерь гарантирует, что профессиональные пользователи могут доверять калькулятору в производственных рабочих процессах и интеграции с 1С.

### III. Соответствие дизайн-системе

Все UI-компоненты ДОЛЖНЫ строго следовать токенам, определённым в `.docs/design-system.json`. Никакие отклонения в цветах, типографике, отступах или стилях компонентов не допускаются.

**Правила**:
- Парсинг и применение каждого токена из design-system.json без модификаций
- Использование CSS-переменных (custom properties), полученных из дизайн-токенов
- Применение соглашения об именовании BEM для всех CSS-классов
- Применение точных значений цветов: background (#f2f4f1), text (#1f2a24), accent (#2f6d4e), и т.д.
- Использование указанной типографики: "Segoe UI", Tahoma, sans-serif; h1: 1.1rem, muted: 0.92rem
- Применение токенов отступов последовательно: xs (6px), sm (8px), md (12px), lg (14px), xl (16px)
- Соответствие стилям компонентов: градиенты кнопок, границы полей ввода, состояния наведения drop-zone

**Обоснование**: Обеспечивает визуальную согласованность, снижает когнитивную нагрузку на пользователей и создаёт профессиональный, полированный вид, соответствующий референсной реализации.

### IV. Модульная архитектура JavaScript

Весь клиентский код ДОЛЖЕН быть организован как ES6-модули со строгим разделением ответственности. Никаких монолитных файлов или spaghetti-кода не допускается.

**Требуемая структура**:
```
assets/js/
├── components/     # UI-компоненты (кнопки, поля ввода, отображение результата, история)
├── logic/          # Основная логика калькулятора, управление состоянием
├── utils/          # Чистые утилитарные функции, помощники
└── app.js          # Главная точка входа (минимальная, только оркестрация)

assets/css/
└── style.css       # Стили со структурой BEM и CSS-переменными
```

**Правила**:
- Каждый модуль имеет единственную, хорошо определённую ответственность
- Все экспортируемые функции включают JSDoc-аннотации типов
- Чистые функции используются везде, где это возможно (без побочных эффектов)
- Каждый модуль независимо тестируется в изоляции
- Никакого загрязнения глобального пространства имён (весь код в модулях или IIFEs)
- Явные импорты/экспорты — никаких неявных зависимостей

**Обоснование**: Модульная архитектура позволяет постепенное тестирование, упрощает отладку, поддерживает командную работу и добавление новых функций без переписывания основных систем.

### V. Лучшие практики WordPress

Плагин ДОЛЖЕН следовать стандартам разработки WordPress для безопасности, совместимости и поддерживаемости.

**Правила**:
- Включить правильный заголовок плагина с именем, версией, описанием, автором, лицензией
- Реализовать хуки активации/деактивации для очистки при необходимости
- Использовать `wp_enqueue_script()` и `wp_enqueue_style()` с правильными зависимостями
- Зарегистрировать шорткод (`[dxf_glc_converter]`) для встраивания калькулятора
- Никакого прямого доступа к файлам — использовать WordPress API где уместно
- Продакшен-билд включает минифицированные ресурсы с хэшами версий

**Обоснование**: Обеспечивает совместимость с экосистемой WordPress, позволяет локализацию для международных пользователей и следует лучшим практикам безопасности, ожидаемым администраторами WordPress-сайтов.

### V.I. Интеграция шорткодов

Калькулятор ДОЛЖЕН быть встраиваемым в любую страницу или публикацию WordPress с помощью простого короткого кода (`[vertexflow_calculator]`). Никакая конфигурация панели администратора, настройки базы данных или хранилище параметров WordPress не разрешены для поведения калькулятора. Вся настройка осуществляется через: 
- Атрибуты шорткода (при необходимости настройки) 
- Инициализация JavaScript на стороне клиента. 
- Прямые ресурсы файловой системы (модули CSS, JS)

**Rationale**: Минимизирует объем базы данных WordPress, позволяет избежать раздувания опций автозагрузки и сохраняет переносимость плагина. Редакторы сайта могут разместить калькулятор где угодно, не пользуясь сложными интерфейсами администратора.

### VI. Качество и производительность

Кодовая база ДОЛЖНА демонстрировать качество профессионального уровня с комплексной обработкой ошибок, поддержкой доступности и оптимизациями производительности.

**Правила**:
- **Обработка ошибок**: Все пользовательские вводы валидируются с понятными сообщениями об ошибках; грациозная деградация при сбоях
- **Доступность (a11y)**: Семантический HTML, ARIA-атрибуты где нужно, поддержка навигации с клавиатуры, достаточный контраст цветов
- **Производительность**: Debounce обработчиков ввода, ленивая загрузка некритичных ресурсов, избежание лишних ре-рендеров
- **Документация**: JSDoc-комментарии ко всем экспортируемым функциям, README.md с инструкциями по установке/использованию
- **Тестирование**: Ручное интеграционное тестирование с реальными GLC-файлами (обязательно); модульные тесты опционально, но рекомендуется
- **Changelog**: Вести CHANGELOG.md с историей версий в формате Keep a Changelog
- **Лицензия**: Включить файл LICENSE (MIT/GPL рекомендуется для WordPress-плагинов)

**Обоснование**: Профессиональные пользователи ожидают надёжные, доступные инструменты. Оптимизации производительности обеспечивают плавный UX во время сложных геометрических операций. Документация снижает нагрузку на поддержку.

## Техническая архитектура

**Платформа**: Плагин WordPress 5.0+ (PHP 7.4+)

**Фронтенд-стек**:
- Vanilla JavaScript (ES6-модули) — без зависимостей фреймворков
- CSS3 с custom properties — без CSS-фреймворков
- SVG-рендеринг для визуализации геометрии
- FileReader API для импорта, Blob/URL API для экспорта

**Поток данных**:
1. Пользователь загружает GLC-файл → FileReader API → GLC-парсер
2. Распарсенные данные → Логика калькулятора → Управление состоянием
3. Изменения состояния → UI-компоненты → Ре-рендеринг с токенами дизайн-системы
4. Пользователь изменяет параметры → Валидация → Пересчёт
5. Запрошен экспорт → GLC-билдер → Blob-скачивание

**Структура директорий**:

## Рабочий процесс разработки

**Стандарты качества кода**:
- Рекомендуется конфигурация ESLint (следовать существующему стилю кода, если не требуется иное)
- Никакого console.log() в продакшен-коде (использовать структурированное логирование при необходимости)
- Никаких глобальных переменных — весь код в модулях или IIFEs
- Согласованное именование: camelCase для переменных/функций, PascalCase для классов, BEM для CSS

**Контроль версий**:
- Семантическое версионирование: MAJOR.MINOR.PATCH
- MAJOR: Критические изменения формата GLC, изменения shortcode API, удалённые функции
- MINOR: Новые функции, новые модули, обратно совместимые улучшения
- PATCH: Исправления ошибок, улучшения производительности, обновления документации

**Чек-лист релиза**:
1. Обновить версию в заголовке плагина
2. Обновить CHANGELOG.md с заметками о релизе
3. Сгенерировать продакшен-билд (минифицированные ресурсы)
4. Протестировать встраивание шорткода на чистой установке WordPress
5. Проверить импорт/экспорт GLC с референсными файлами из `.docs/GLC_FORMAL_SPECIFICATION/`
6. Валидировать соответствие дизайн-системе (рекомендуется визуальное регрессионное тестирование)
